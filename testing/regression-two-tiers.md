# 회귀 확인은 "자동 게이트"와 "사람 판단"을 나눈다

파이프라인/프롬프트를 바꿀 때마다 전체 데이터셋을 다시 돌려 결과를 비교하는 건
느리고(수십 분+) 노이즈도 많다. 그래서 회귀 확인을 두 층위로 나눴다.

## 두 층위

1. **빠른 회귀 세트 (개발 반복용, 고정 소수)**
   전량 대신 대표 케이스만 고정 목록으로 동결해 빠르게 돌린다. 목록을 코드로
   박지 않고 manifest 파일에 얼려두면, "왜 이 케이스를 골랐나"가 이력으로 남고
   재선정도 의도적으로만 일어난다.

2. **전량 테스트 (야간/CI에 맡김)**
   시간이 오래 걸리는 전체 비교는 사람이 실시간으로 지켜보지 않는 시간대에
   돌린다. 빠른 세트에서 회귀가 없음을 확인하고, 넓은 커버리지는 이쪽에 맡긴다.

## 핵심: 신호를 두 종류로 갈랐다

빠른 세트 안에서도 케이스마다 **자동으로 판정 가능한 것**과 **사람이 봐야 하는 것**을
구분한 게 제일 중요했다.

- **자동 게이트 가능** — 신뢰할 수 있는 정답이 있는 케이스는 점수(예: F1) 델타로
  판정한다. 하락하면 그 자체가 회귀 신호다. 사람이 볼 필요 없다.
- **사람 판단 필요** — 정답이 확정되지 않은 케이스는 점수로 못 가른다. 대신
  이전 실행 대비 **무엇이 바뀌었는지 diff만 표면화**하고, 개선인지 회귀인지는
  사람이 본다. 단일 필드 하나 뒤집힌 건 노이즈에 가깝고, 항목이 통째로
  추가/삭제된 건 강한 신호 — 이렇게 변화의 크기로 주의를 배분한다.

## 배운 것

"회귀 테스트"를 한 덩어리로 두면 느리거나(전량) 신뢰 못 할(정답 없는 걸 점수로
우김) 둘 중 하나가 된다. **정답의 신뢰도**를 기준으로 자동 게이트와 사람 검토를
분리하고, 실행 비용을 기준으로 빠른 세트와 전량을 분리하면, 개발 반복은 빠르게
돌면서도 놓치는 회귀는 야간 전량이 잡아준다. diff는 "판정"이 아니라 "주의를
어디로 돌릴지"를 정하는 도구로 쓴다.
